§ 01

What we are building, and why.

A production-grade Agentic RAG system that answers threat-intelligence questions over publicly available cybersecurity corpora — built end-to-end with the engineering discipline of a real platform team, not a tutorial project.

The system ingests heterogeneous security data (CVE/NVD records, MITRE ATT&CK techniques, vendor advisories, public threat-actor reports, detection rules), indexes it into a hybrid retrieval layer, and exposes a stateful agentic query engine via an authenticated HTTP API. The agent reasons about the question, decides which sources to query, performs iterative retrieval with self-grading, and produces cited, auditable answers within enforced time and cost budgets.

Every architectural decision in this brief is made to be defensible and explicitly documented — trade-offs acknowledged, alternatives considered, and rationale recorded.

§ 02

Subject matter & corpora.

Cybersecurity threat intelligence is chosen because it surfaces every interesting RAG engineering problem: heterogeneous source schemas, heavy metadata filtering, citation-mandatory output, time-sensitivity, and multi-hop reasoning across structured and unstructured data. The corpora are all publicly accessible.

  • NVD & CVE Records
    services.nvd.nist.gov
    Structured vulnerability records with CVSS scores, affected products (CPE), and free-text descriptions. The "structured-meets-unstructured" backbone.
  • MITRE ATT&CK
    attack.mitre.org · STIX/TAXII
    Tactics, techniques, sub-techniques, threat-actor profiles, and software catalogs. Graph-shaped; introduces relationship-aware retrieval.
  • Vendor Advisories
    Microsoft MSRC · Cisco PSIRT · Red Hat
    Long-form narrative advisories with embedded IOCs and remediation guidance. Forces table-aware and section-aware chunking.
  • Detection Rule Repositories
    SigmaHQ · YARA-Rules (GitHub)
    YAML/regex rules mapped to ATT&CK techniques. Introduces cross-corpus joining: "what detects this TTP?"
  • Public Threat-Actor Reports
    Mandiant · Unit 42 · CrowdStrike (public)
    Narrative attribution and campaign analysis. The hardest unstructured tier; tests entity linking and temporal reasoning.

Example Queries the System Must Handle

"What recent vulnerabilities affect Kubernetes 1.28, have public proofs-of-concept, and which detection rules cover them?"
"What TTPs has APT29 used against financial services in the last 24 months, and how do they map to MITRE ATT&CK?"
"Compare the remediation guidance across the latest Microsoft Exchange advisories and identify common attacker techniques."

Each example forces multi-source retrieval, temporal filtering, and cross-corpus reasoning. A single vector search cannot answer any of them — which is precisely the point.

§ 04

LLM provider strategy.

Start local with Ollama. Design provider-agnostic from day one. Earn the right to spend tokens.

Development begins with Ollama running locally as the LLM provider. The starting model is llama3.1:8b-instruct for generation and nomic-embed-text for embeddings — both run on CPU or a modest GPU, both produce sane output for development iteration, and both cost zero tokens. We expect quality to be visibly worse than a frontier API model; that is acceptable and arguably useful, because hard prompt-design and retrieval problems show up earlier when the model is weaker.

The architectural commitment is to a LLMProvider abstraction from the very first commit: a Protocol class with generate(), stream(), and embed() methods, and concrete implementations for Ollama and (later) Anthropic, OpenAI, and any other provider. The agent loop, retrieval layer, and API surface depend only on the Protocol — never on a concrete provider.

Cost Discipline

Frontier-model API calls are introduced only when (a) we have an evaluation harness to measure the quality delta, and (b) we have a defensible reason — usually that the agent's planning step needs better reasoning than the local model provides. Every API call must be logged with cost in Langfuse.

§ 05

Technology stack & justifications.

Each choice below is justified on technical merit. Where there is a genuine trade-off, the alternative and the reason for rejection are stated. Every component must earn its place.

FastAPI
HTTP API Layer
Async-native, Pydantic v2 schemas double as agent structured-output contracts, SSE streaming via StreamingResponse, auto-generated OpenAPI. Behind Uvicorn workers with CPU-bound work delegated to Celery so the event loop never blocks.
Alt: Litestar (lighter, faster routing) or Go (best throughput). FastAPI wins on ecosystem and team familiarity.
RabbitMQ + Celery
Async Task Queue
Durable broker with proper routing, DLQs, and priority queues for ingestion, scheduled feed pulls (Celery Beat), long agent runs, and eval execution. Boring, well-understood, production-proven.
Alt: Temporal would be superior for the long agent-run workflow specifically (durable execution, native retries) — acknowledged trade-off, may revisit in Phase 6+.
LlamaIndex
Retrieval Primitives
Document parsing (readers), hierarchical and sentence-window indexers, query engines, NodePostprocessors for reranking. Owns the "data → vectors → ranked results" half of the system.
Alt: LangChain retrievers — LlamaIndex has more mature hierarchical indexing and post-processors for this workload.
LangGraph
Agent Orchestration
Explicit state-machine graph for the agent loop (route → retrieve → grade → refine → generate → reflect). Checkpointing enables resumable runs; nodes are individually inspectable and traceable.
Alt: Hand-rolled state machine — viable but loses checkpointing and interrupt patterns we'll want for human-in-the-loop later.
Qdrant
Vector Database
Pre-filtered HNSW search via payload indexes — decisive for threat intel's heavy metadata filtering (CVSS, dates, vendor, severity). Native sparse + dense hybrid via named vectors. First-class quantization and multi-tenancy patterns.
Alt: pgvector — simpler ops, one Postgres; acceptable at smaller scale, especially with pgvectorscale. Weaviate — heavier, more opinionated. Chose Qdrant for filter performance and sparse + dense hybrid support.
PostgreSQL
Relational State
Tenant data, API keys, ingestion CDC table (tracks "what's been indexed and at what embedding version"), agent-run records, audit log. The system of record for everything that isn't a vector.
Not optional. Vector DB is not a replacement for relational state.
Redis
Cache & Rate Limit Store
Semantic and exact response caching, token-bucket rate limit counters, Celery result backend (optional), idempotency-key store. Sub-millisecond access patterns the agent loop needs.
Could use Postgres for some of these; Redis is correct on latency grounds for the hot path.
Langfuse
LLM Observability
Full traces of every prompt, completion, tool call, and retrieval — with cost and latency per node. Prompt versioning, dataset management, eval runs. Self-hosted via Docker for data control.
Alt: OpenTelemetry + custom span attributes — viable but loses LLM-specific UX (prompt diffing, cost rollups). Use both: Langfuse for LLM ops, OTel for infra.
Ollama
Local LLM Runtime
Local-first development. Zero per-token cost during the long build phase. Forces robust prompts and retrieval — weak models surface bad design earlier than strong ones hide it.
Quality ceiling is real. Provider abstraction allows swapping to Anthropic/OpenAI per-stage when justified by evals.
Pydantic v2 + Pydantic Settings
Validation & Config
Single source of truth for data shapes (API I/O, agent structured outputs, ingestion records) and environment configuration. Rust-backed validators are fast enough to use everywhere.
Non-negotiable for a Python AI backend in 2026.

§ 07

Development phases.

The build is sequenced so that every later phase rests on a verified earlier one. We do not move forward until the current phase's verification criteria pass. Skipping the "boring" early work is the most common way projects like this collapse.

PHASE
00

Foundations

Repo scaffolding with uv for dependency management. compose.yaml for Postgres, Qdrant, RabbitMQ, Redis, Ollama, Langfuse. Pydantic Settings for env config. Structured logging (structlog). FastAPI skeleton with health/ready/live endpoints. Pre-commit hooks (ruff, mypy, prettier). GitHub Actions CI scaffold. NOTES.md and a starter README.md.

DELIVERABLE · `docker compose up` brings the full stack to green health
PHASE
01

Ingestion & Indexing

Single-source ingestion pipeline end-to-end against the NVD/CVE feed. Celery worker fetches, normalizes to a Pydantic schema, content-hashes for idempotency, chunks via LlamaIndex, embeds with Ollama's nomic-embed-text, upserts to Qdrant with full payload metadata. Postgres CDC table tracks ingestion state and embedding-model version. Manual retrieval queries via a CLI to verify quality before any API exists.

DELIVERABLE · Queryable Qdrant index of ≥10k CVE records with payload filters
PHASE
02

Naive RAG Baseline

Simplest viable query → embed → top-k retrieve → stuff prompt → generate. Wrapped in a FastAPI endpoint with SSE streaming. The point is not quality — it is to establish a baseline against which every later improvement is measured. We hand-curate a 30–50 question golden set at this phase and record baseline scores.

DELIVERABLE · `POST /v1/query` returns streamed answers · baseline eval recorded
PHASE
03

Retrieval Quality

Hybrid retrieval (sparse + dense via Qdrant named vectors). Cross-encoder reranking via a local sentence-transformers model. Query rewriting and decomposition. Each addition is measured against the baseline — the eval table grows row by row.

DELIVERABLE · Measured improvements in faithfulness and context recall vs. baseline
PHASE
04

The Agentic Layer

LangGraph state machine: ROUTE → RETRIEVE → GRADE → REFINE → GENERATE → REFLECT. Tool registry (vector retrieval, SQL over CVE metadata, optional web fresh). Budget enforcement — hard caps on max steps, tokens, cost. Structured outputs via Pydantic. Persistent agent state with checkpointing.

DELIVERABLE · Multi-hop queries succeed where naive RAG fails · all traces inspectable
PHASE
05

Observability & Evaluation

Full Langfuse integration — every node, prompt, completion, tool call traced with cost and latency. Prompt versioning. Golden-set evaluation runs in CI on every merge. Regression detection that fails the build on quality drops. Custom dashboards for cost per query and end-to-end latency budget breakdown.

DELIVERABLE · CI fails on eval regression · Langfuse dashboards documented in README
PHASE
06

Production Hardening

Per-tenant rate limiting with token-bucket in Redis. Multi-tenant isolation in Qdrant (collection-per-tenant) and Postgres (row-level). OAuth2 with scoped bearer tokens. Idempotency keys. RFC 7807 problem-details. Async agent-run resource pattern (POST returns 202 + run ID; GET polls or webhook callback). OpenTelemetry traces. Prometheus /metrics. Graceful shutdown.

DELIVERABLE · Two-tenant load test passes with isolation verified
PHASE
07

Testing & Optimization

Pytest with async fixtures. Integration tests via testcontainers. Load tests with k6 producing the latency chart in the README. Profiling and bottleneck analysis. Semantic caching layer. Embedding-model upgrade path exercised end-to-end. The eval-results table and load-test chart are finalized as project deliverables.

DELIVERABLE · p95 latency chart · cost-per-query dashboard · ≥80% test coverage
PHASE
08

Documentation & Polish

Final README rewrite with problem statement, measurable outcomes, architecture diagram, and one-command demo. Recorded video walkthrough or live deployment. Incident postmortem entry describing one real failure encountered and fixed.

DELIVERABLE · Documentation complete · one-command demo working end-to-end
§ 08

Development methodology.

Each phase is built incrementally. Every step is verified before moving forward. The commit history documents the build progression as a sequence of working states.

The approach for each step is consistent: understand the rationale before writing code, implement against a clear specification, verify with the exact command and expected output, then commit. No step advances until its verification passes.

Incremental delivery

Three phases done properly — with real verification criteria, clean commits, and documented trade-offs — produce more value than eight phases done superficially. The phase sequencing in §07 reflects this: each phase's deliverable is a concrete, runnable artifact, not just merged code.

§ 09

Ground rules.

  1. Commit after every working step. Each verified step is its own commit, with a clear message. By the end the git history reads as a complete development journal. Do not squash.
  2. Maintain NOTES.md in the repo. A running log of what was learned, what surprised, and what would be done differently. Invaluable for understanding decisions made months later. Raw material for future blog posts.
  3. Do not skip "boring" steps. Pre-commit hooks, CI from day one, structured logging, type checking — these are standard engineering discipline, not optional extras. If a step feels skippable, surface it explicitly before deferring.
  4. Stop when a concept is unclear. "Wait, why does HNSW need pre-filtering specifically?" is the right kind of question. Better to understand the concept properly before building on it than to paper over a gap that surfaces later.
  5. Realistic pace. 60–100 hours of focused work over 6–10 weeks at a few evenings per week. Compression to a weekend produces copy-paste. Three phases done properly outperform eight done superficially.
  6. When frustrated, say so. Frustration is a signal to slow down, change angle of explanation, or simplify the step — not to heroically push through and ChatGPT the answer. The goal is understanding, not endurance.
§ 10

Resumption protocol.

This document is the single source of truth for the project. To re-orient after any interruption, check three things: the current phase status in CLAUDE.md, the most recent commit in git log, and any open notes in NOTES.md.

Orientation Checklist

current_phase: check CLAUDE.md phase status
last_commit:   git log --oneline -5
service_health: docker compose ps
open_items:    check NOTES.md for unresolved questions
State invariant

Every phase ends with a passing verification step and a clean commit. If a phase is in progress, the last clean commit marks where verified state ends — resume from there, not from unverified working-directory changes.
